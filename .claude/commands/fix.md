# /fix - バグ修正コマンド

## 目的

報告されたバグを調査し、修正するワークフローを実行します。

## ワークフロー

```
┌─────────────────────────────────────────────────────────────┐
│  Phase 1: 問題分析                                          │
│  Orchestrator がバグを分析、担当を決定                       │
├─────────────────────────────────────────────────────────────┤
│  Phase 2: 原因調査                                          │
│  Worker が原因を特定                                        │
├─────────────────────────────────────────────────────────────┤
│  Phase 3: 修正実装                                          │
│  Worker が修正とテストを作成                                 │
├─────────────────────────────────────────────────────────────┤
│  Phase 4: 検証                                              │
│  Tester がリグレッションテスト                               │
├─────────────────────────────────────────────────────────────┤
│  Phase 5: レビュー                                          │
│  Reviewer が修正内容を確認                                   │
├─────────────────────────────────────────────────────────────┤
│  Phase 6: 完了                                              │
│  Orchestrator が最終確認、CHANGELOG更新                     │
└─────────────────────────────────────────────────────────────┘
```

## 緊急度の判断

| 緊急度 | 条件 | 対応 |
|--------|------|------|
| Critical | 本番障害、データ損失 | 即時対応、最優先 |
| High | 主要機能が使用不可 | 当日中に対応 |
| Medium | 機能制限、回避策あり | 計画的に対応 |
| Low | 軽微、UX改善 | バックログ |

## 関連コマンド

- `/develop` - 機能開発フロー
- `/refactor` - リファクタリングフロー
- `/improve` - システム改善フロー
