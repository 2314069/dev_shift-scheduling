# /develop - 機能開発コマンド

## 目的

新機能を開発するための包括的なワークフローを実行します。
ハイブリッド型マルチエージェント構成で、品質と効率を両立します。

## ワークフロー

```
┌─────────────────────────────────────────────────────────────┐
│  Phase 1: 分析・計画                                        │
│  Orchestrator がタスクを分解                                │
├─────────────────────────────────────────────────────────────┤
│  Phase 2: 設計                                              │
│  Planner → Designer （直列）                                │
├─────────────────────────────────────────────────────────────┤
│  Phase 3: 実装                                              │
│  Worker × N （並列）                                        │
├─────────────────────────────────────────────────────────────┤
│  Phase 4: テスト                                            │
│  Tester （統合テスト・E2E）                                  │
├─────────────────────────────────────────────────────────────┤
│  Phase 5: 仕上げ                                            │
│  Reviewer + Documenter （並列）                             │
├─────────────────────────────────────────────────────────────┤
│  Phase 6: 完了                                              │
│  Orchestrator が最終確認・統合                              │
└─────────────────────────────────────────────────────────────┘
```

## 実行手順

### Phase 1: 分析・計画（Orchestrator）
1. ユーザーの要件を分析
2. タスクを独立した単位に分解
3. 依存関係を特定、並列化可能な部分を決定

### Phase 2: 設計
- **2.1 技術設計（Planner）**: 要件定義、アーキテクチャ、DB、API設計
- **2.2 UI/UX設計（Designer）**: 画面設計、コンポーネント設計、スタイルガイド

### Phase 3: 実装（Worker × N）
各 Worker が独立した機能を並列で実装（コード + 単体テスト）

### Phase 4: テスト（Tester）
統合テスト・E2Eテストの作成・実行、問題の報告

### Phase 5: 仕上げ（並列）
- **5.1 コードレビュー（Reviewer）**: 品質・セキュリティレビュー
- **5.2 ドキュメント作成（Documenter）**: README更新、使い方ガイド、CHANGELOG

### Phase 6: 完了（Orchestrator）
全成果物の確認、品質基準の最終チェック、完了報告

## 注意事項

- 各フェーズ完了後、Orchestrator が品質を確認してから次へ進む
- 問題が発見された場合、該当エージェントに差し戻し
- 大規模な機能は複数回に分けて実行を推奨

## 関連コマンド

- `/fix` - バグ修正フロー
- `/refactor` - リファクタリングフロー
- `/improve` - システム改善フロー
